## 你的 Token 钱，有一半是白花的

用 Claude Code、Cursor、Windsurf 这类 AI 编码工具的同学，每个月看 API 账单的时候应该都肉疼过。

每次 AI 回你一段话，前面都跟着一大段「当然！我很乐意帮助你解决这个问题……」之类的废话。代码本身可能就 20 行，但前面铺垫了 100 个 token 的「客气话」，后面又跟了 50 个 token 的总结建议。

这些废话的学名叫 **filler tokens**，它们不产生任何实际价值，但每一条都按字数收费。

最近 GitHub 上有个项目火得一塌糊涂，**从零到 8.5 万星只用了不到一个月**。它做的事情简单到离谱——让 AI 说话像原始人，然后省掉 65% 的 Token。

项目名字就叫 **Caveman**。

## Caveman 是什么？

> 直译是「山顶洞人」，但更贴切的翻译是「话少说，活照干」。

Caveman 是一个为 AI 编码助手（Claude Code、Codex、Gemini CLI、Cursor、Copilot 等 30+ 工具）设计的技能/插件。它的核心逻辑只有一句话：

**不改变 AI 知道什么，只改变 AI 说什么。**

所以你得到的结果是：
- 代码质量不变（技术内容 100% 保留）
- 废话全部砍掉（输出 Token 减少 65%）
- 阅读效率翻倍（一眼看重点）

## 安装方式：一行命令

### macOS / Linux / WSL
```bash
curl -fsSL https://raw.githubusercontent.com/JuliusBrussee/caveman/main/install.sh | bash
```

### Windows PowerShell
```powershell
irm https://raw.githubusercontent.com/JuliusBrussee/caveman/main/install.ps1 | iex
```

安装过程大约 30 秒，需要 Node ≥ 18，会自动检测你机器上已有的 AI 工具并安装。

如果你只用 Claude Code，也可以走插件市场：
```bash
claude plugin marketplace add JuliusBrussee/caveman
claude plugin install caveman@caveman
```

## 效果到底怎么样？直接看对比

### 场景一：解释 React 重渲染问题

**正常 AI 回复**（69 tokens）：
> "The reason your React component is re-rendering is likely because you're creating a new object reference on each render cycle. When you pass an inline object as a prop, React's shallow comparison sees it as a different object every time, which triggers a re-render. I'd recommend using useMemo to memoize the object."

**Caveman 回复**（19 tokens）：
> "New object ref each render. Inline object prop = new ref = re-render. Wrap in useMemo."

Token 节省：**87%**。信息量完全相同。

### 场景二：修复 Auth 中间件 Token 过期 Bug

**正常 AI 回复**：
> "Sure! I'd be happy to help you with that. The issue you're experiencing is most likely caused by your authentication middleware not properly validating the token expiry. Let me take a look and suggest a fix."

**Caveman 回复**：
> "Bug in auth middleware. Token expiry check use < not <=. Fix:"

83% 的废话被去掉了，直接告诉你问题在哪、怎么修。

## 我的观点：为什么这个项目会火

Caveman 的爆火不是偶然，它精准地戳中了三个痛点：

### 1. AI 行业在「祛魅」
2025 年大家还在惊叹 AI 能写代码。到了 2026 年，开发者早已过了新鲜期，开始精打细算。每个月花几百上千刀在 API 上，换回来的却是大段大段的废话——没人愿意当这个冤大头。

### 2. 阅读效率比 Token 成本更重要
Caveman 最大的价值未必是省钱。想想你每天读多少 AI 输出的文本——如果每段能省 65% 的文字，**你的注意力就是最大受益者**。开发者一天要扫无数行回复，废话少意味着扫读更快，决策更快。

### 3. 一个有趣的研究背书
有一篇 2026 年 3 月的论文《Brevity Constraints Reverse Performance Hierarchies in Language Models》发现：**对大模型施加简洁约束后，准确率反而提升了约 26 个百分点**。也就是说，让 AI 说人话比让 AI 说「好听的废话」更容易得到正确答案。

## 使用技巧

### 六种模式切换

Caveman 内置了 6 级压缩强度，随时可调：

```
/caveman lite     # 轻度压缩
/caveman full     # 默认，中度压缩
/caveman ultra    # 极限压缩
/caveman wenyan   # 文言文模式（中文压缩率最高）
```

我个人推荐 **full 模式**，既保留了足够的信息完整性，阅读体验也最好。

### 实用命令

| 命令 | 作用 |
|------|------|
| `/caveman-stats` | 查看当前会话节省的 Token 数 |
| `/caveman-commit` | 生成 ≤50 字 Conventional Commit |
| `/caveman-review` | 单行 PR Review 评论 |
| `/caveman-compress <file>` | 把你的记忆文件压缩成 Caveman 格式 |

其中 `caveman-compress` 是个隐藏神器——把 CLAUDE.md 这类项目说明压缩 46% 后，每次会话加载都少读一半的 input token，**长期省钱效果比压缩 output 更明显**。

## 老实说，也有缺点

Caveman 不是全能的。作者自己在文档里也坦诚了几点：

1. **Skill 本身会消耗 ~1-1.5k input token/次**，在某些对话场景下可能净收益为负
2. **对已经说得很短的 AI（比如 Codex），节省空间有限**
3. **团队协作时可能不太合适**——如果你的队友需要看 AI 的完整推理过程，Caveman 的简洁风格反而会增加沟通成本
4. 如果遇到复杂问题，ultra 模式可能会过于简略导致信息丢失

所以我的建议是：日常编码用 full 模式，遇到复杂架构讨论时切回 normal。灵活切换才是正解。

## 结语

Caveman 给我最大的启发不是省了多少钱，而是重新定义了一个问题：**AI 的输出价值是什么？**

如果 65% 的 Token 是废话，那说明整个行业的 prompt engineering 方向都有问题——我们一直在教 AI 怎么「礼貌地回答问题」，而不是怎么「高效地解决问题」。Caveman 反其道而行，效果出奇的好。

就像项目介绍里那句最好笑也最扎心的话：

> **Why use many token when few token do trick.**
