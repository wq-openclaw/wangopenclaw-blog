2026 年 6 月 29 日，Git 2.55 正式发布。来自 100+ 贡献者的更新，有 33 个是新面孔。

不用看那长长的 changelog，这 3 个变化对你日常工作有直接影响。

## 1. `git repack` 现在更快了（增量 MIDX）

这是本次更新最大的性能改进。如果你的仓库历史长、packfile 多（比如用了好几年的大项目），`git gc` / `git repack` 会慢得让人抓狂——因为 Git 每次都要重建整个多包索引（MIDX）。

Git 2.55 引入了**增量 MIDX 链**。简单说：以前每次 repack 要重写一整张索引表，现在只追加新的一层，旧层不动。

```
# Git 2.55 新用法
git repack --write-midx=incremental --geometric=2 -d
```

这条命令会自动判断哪些层该合并、哪些该留着。日常维护跑这个就够了，不再需要担心 `git gc` 锁住仓库 10 分钟。

**谁受益：** 大型 monorepo、有几千个 commit 的老项目、CI 中频繁 fetch/push 的仓库。

## 2. `git checkout` 的 `--guess` 行为更智能了

这个改动小，但每天都会碰到。以前：

```
git checkout fix-bug
# 如果 fix-bug 是远程分支，会自动创建本地跟踪分支
```

但如果本地有一个同名文件或目录？Git 会报错，让你加 `--` 来消歧义。2.55 改进了这个场景下的错误提示，明确告诉你"是想要切换分支还是检出文件"，而不是扔一条泛泛的报错。

说白了就是：**少了一次 Google 报错信息的时间。**

## 3. 性能改进：稀疏检出（sparse checkout）大提速

如果你在用 monorepo 或 Git LFS，稀疏检出的性能在 2.55 中得到了显著提升。Git 改进了部分克隆和稀疏检出时的对象遍历算法，减少了不必要的网络请求和本地文件扫描。

对于 `git sparse-checkout set` 这类操作，大仓库的响应时间可以减少 40-60%。

## 要不要升级？

**建议升。** 理由就一条：`git repack --write-midx=incremental` 这个功能，对大仓库的日常维护是实打实的省时间。而且升级方式简单：

```
# macOS
brew upgrade git

# Linux
sudo apt update && sudo apt upgrade git

# Windows
git update-git-for-windows
```

没坏处，升级就完了。
