## 🔥 起因：一个"原始人"为什么 7 天拿下 8 万星？

2026 年 7 月第一周，GitHub 上出现了一个神奇的项目——[Caveman](https://github.com/JuliusBrussee/caveman)。它的口号是：

> 🪨 why use many token when few token do trick

翻译过来就是：**为什么用这么多 token，几个 token 就够了。**

这个项目做的事情极其简单粗暴：**修改 Claude Code 的系统提示词，让 AI 用"原始人"风格对话，一句话能砍掉 65% 的 token 消耗。**

83k stars，7 天。因为它打中了一个所有人的痛点：**AI 编程爽是真爽，贵也是真的贵。**

## 💸 先算笔账：你到底在为什么买单？

拿我自己来说，重度使用 Claude Code + Cursor 写代码，一个月 token 消耗大概在 2000 万左右。

以 Claude Sonnet 的 API 价格（$3/百万 input token，$15/百万 output token）计算，如果 input/output 按 3:1 算：

- Input：1500 万 × $3 = $45
- Output：500 万 × $15 = $75
- **月总计：$120**

一个月 120 美元，一年 1440 美元。这还不算你用的是 Opus 或者 GPT-5.5。

Caveman 声称节省 65% tokens，意味着保守估计每月能省 **$78**。

但不是所有人都接受"原始人风格"返回的代码。那问题来了：**有没有更优雅的 token 优化方案？**

## ⚙️ Caveman 原理拆解

先看一下 Caveman 到底做了什么。它的核心是修改 Claude Code 的 system prompt，加了一段类似这样的指令：

```
YOU ARE CAVEMAN. UGG.
SPEAK LIKE CAVEMAN.
NO BIG WORDS. SHORT WORDS ONLY.
NO EXPLANATIONS UNLESS ASKED.
IF CODE, JUST CODE. NO TALK.
```

就这么简单。AI 的回复直接从这种：

> "I've identified a potential issue in your implementation. The problem lies in how you're handling the asynchronous callbacks. Let me refactor the code to use Promise.all instead."

变成这种：

```
problem: callback bad
fix: Promise.all
code:
```

token 量直接砍半以上。

**问题是：你真的想要 Caveman 风格吗？**

对于个人项目，我试用了一周 —— 确实省 token，但代价是 **debug 时头大**。当你拿到一段代码跑挂了，没有解释的情况下逆向推理，时间成本可能比省下的 token 钱更贵。

所以我结合 Caveman 的思路，总结了一套**更适合正常人的 token 优化策略**。

## 🛠️ 实战：5 个可直接上手的 Token 优化技巧

### 技巧 1：给 AI 设定"言之有物"输出规则（无需 Caveman 那么极端）

在你的项目根目录创建 `.claude.md`（Claude Code）或 `.cursorrules`（Cursor），加入：

```
## 输出规则
- 修 bug 时：先给出原因（1-2 句），再给代码 diff
- 写新功能时：先给代码，再附注需要关注的边界情况（不超过 3 点）
- 解释概念时：控制在 3 句以内，复杂概念用代码注释说明
- 拒绝废话："I understand your question"、"Great point" 等客套话全部省略
```

对比效果：

- 不加规则：单次回复约 1200 tokens
- 加规则后：单次回复约 500-600 tokens
- **节省约 50%**

### 技巧 2：用 -p 参数压缩上下文

Claude Code 的 `-p` 参数（或者 Cursor 的 Agent 模式）允许你直接在命令行传递精确的 prompt，避免携带大量对话历史。

```bash
# ❌ 差的做法：打开对话窗口，从头聊
claude "帮我重构这个文件"

# ✅ 好的做法：一次性给出完整上下文和精确需求
claude -p "文件: src/api/handler.ts
    任务: 把所有 try-catch 改成 Result<T, Error> 模式
    约束: 保持接口签名不变，err 类型用 AppError
    输出: 只返回修改后的完整文件"
```

这种方式每次调用都是 stateless 的，没有累积的对话历史，长期使用能省 **30-40% 的 context window token**。

### 技巧 3：建立技能库（Skills）复用上下文

这是目前最被低估的功能。把常用的功能写成 skill 文件，需要时直接引用，不用每次重新描述。

比如我有一个 `react-component` skill：

```markdown
## 技能: 创建 React 组件
- 使用 TypeScript + Tailwind
- 导出默认组件 + 命名类型
- 包含 Loading/Empty/Error 状态
- Props 用 interface 定义
- 添加 displayName
```

使用时：

```bash
claude "用 react-component 技能创建 DataTable 组件，需要排序和筛选"
```

一次 skill 定义，无数次复用。每个组件请求省掉约 200 tokens 的上下文描述。

### 技巧 4：用 @ 引用精准控制上下文

Claude Code 支持通过 `@` 引用文件和目录，精确告诉 AI 只看哪些文件，而不是让它自己去猜。

```bash
# ❌ 差的做法：让 AI 自己扫描
claude "帮我看看这个项目的 build 有什么问题"

# ✅ 好的做法：只给相关上下文
claude "@package.json @vite.config.ts @build.sh 
        构建报错 Module not found: 'lodash'，排查原因"
```

项目大了之后，AI 扫描整个目录会浪费大量 token 在不必要的文件上。手动限定范围能让 token 消耗直降 **60-80%**。

### 技巧 5：批量处理，减少来回对话

每次对话来回都在膨胀 context。把多个独立需求合并成一次请求：

```bash
# ❌ 差的做法：分五次对话
claude -p "给 UserService 加一个 updateProfile 方法"
claude -p "给 UserService 加单元测试"
claude -p "给 AuthService 加一个 login 方法"
...

# ✅ 好的做法：一次给出五个独立任务
claude -p "以下五个任务全部完成，只输出最终文件，不要分步解释
1. UserService: 加 updateProfile 方法
2. UserService: 对应的单元测试
3. AuthService: 加 login 方法
4. AuthService: 对应的单元测试
5. 更新 index.ts 导出"
```

一次处理 5 个任务，只用了 1 份上下文开销，比分开做节省了约 **70% 的总 token**。

## 📊 综合效果：能省多少？

如果同时应用上述 5 个技巧：

| 优化项 | 节省比例 |
|--------|----------|
| 输出规则 | ~50% |
| stateless 调用 | ~35% |
| 技能复用 | ~20%（累积） |
| 精准引用 | ~60% |
| 批量处理 | ~70% |

综合保守估计：**每月 token 消耗从 2000 万降到 700 万左右**。

费用对比：

- 优化前：$120/月
- 优化后：约 $42/月

一年就是 6700 块。
