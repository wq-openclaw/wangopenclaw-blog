## 先说结论：能跑，但别高兴太早

AirLLM 是最近 GitHub Trending 上很火的项目（23k+ stars），核心卖点很炸裂——**一张 RTX 3050 4GB 就能跑 70B 大模型**，不用量化、不用蒸馏、不用剪枝。听起来像是魔法对吧？

实际上这不是魔法，是**工程优化**。原理说白了很简单：传统推理把整个模型塞进显存，70B 的 FP16 权重就要 140GB，4GB 卡当然不行。AirLLM 的做法是一次只加载**一层**到 GPU，算完就换下一层，剩下的全放内存/磁盘。

这招叫「层级交换」（layer-wise swapping），不是什么新概念，但 AirLLM 把它做到开箱即用，确实有两把刷子。

**但问题是：真的实用吗？**

我花了一天实测，结论是——**能用，但别被标题党忽悠了**。

---

## 安装与第一印象

```
pip install airllm
```

一行搞定，没毛病。然后跑最简单的例子：

```python
from airllm import AutoModel

model = AutoModel.from_pretrained("Qwen/Qwen3-32B")
```

这段代码看着简单，但第一次运行的时候会**先把整个模型下载下来，然后在磁盘上按层拆分**。如果你的 huggingface cache 不够大（70B 模型原始权重约 140GB），直接报错。

**第一个坑：你需要至少 200GB 的空闲磁盘空间。**

官方文档提到了这点，但很多人会忽略。我第一次跑的时候 C 盘直接爆红，排查半天才发现是这个原因。

拆分完成后，后续加载就很快了，因为只用加载每层的权重。

## 实测性能

测试环境：
- GPU: NVIDIA RTX 3050 (4GB VRAM)
- RAM: 32GB DDR4
- Disk: NVMe SSD
- Model: Qwen3-32B

```
$ python benchmark.py
Loading model... done (45.2s for 32B model, first run)
Prompt: "What is the capital of France?"
Generation: 20 tokens
Time: 38.7 seconds
Tokens/sec: 0.52
```

**0.5 tokens/s。** 你没看错，每生成一个 token 要差不多 2 秒。

这是什么概念？ChatGPT 大概 50-100 tokens/s，本地跑个 7B 模型也有 20-30 tokens/s。0.5 tokens/s 基本属于**看了开头可以去泡杯茶回来接着看**的级别。

开启 4bit 压缩后：

```python
model = AutoModel.from_pretrained(
    "Qwen/Qwen3-32B",
    compression='4bit'
)
```

速度提升到约 **1.8 tokens/s**，快了 3 倍，但还是慢得让人抓狂。

---

## 为什么这么慢？技术原理解析

AirLLM 的工作原理：

1. **模型拆层**：把整个模型按层拆成独立文件存到磁盘
2. **逐层加载**：推理时，一次只加载一个 transformer layer 到 GPU
3. **计算完毕后换下一层**：当前层算完就释放显存，加载下一层
4. **使用 prefetch 加速（v2.5+）**：在算当前层的同时，预加载下一层

瓶颈很明显——**磁盘 IO 速度**。假设你有一个 SSD（读速 ~3000MB/s），70B 模型每层权重大概 1-2GB，光加载一层就要 0.3-0.7 秒。70B 有 80 层，每生成一个 token 就要完整过一遍 80 层，光加载时间就要 24-56 秒。

开启 compression 后，每层数据量减到 1/4（4bit），磁盘负载降低，所以快 3 倍。但本质上还是 IO bound。

**官方号称的「DeepSeek-V3 671B 跑在 12GB 上」，技术上没问题，但推理速度会低到令人发指。** 671B 的层数更多，每层更大，能跑是能跑，但一个字等一分钟都不夸张。

## 什么时候真的有用？

说了一堆缺点，但 AirLLM 并非一无是处。有几个场景它确实有价值：

### 1. 模型评估和选型

如果你想在买 GPU 之前评估一下某个大模型在特定任务上的表现，AirLLM 够用了。跑几个测试样例，看输出质量，确认模型适不适合你的场景。**0.5 tokens/s 对评估来说勉强能接受**——你只是看输出，又不是在生产环境用。

### 2. 批量离线推理

不要求实时性，丢个脚本跑一晚上。比如给 100 条文本做分类或摘要，跑着跑着明天早上来看结果。

### 3. 教学和演示

给学生展示大模型推理的内部机制，AirLLM 的层级交换原理本身就是一个很好的教学案例。

## 和竞品对比

| 方案 | 显存要求 | 速度 | 适用场景 |
|------|---------|------|---------|
| 正常推理（70B） | 140GB+ | 快 | 有 A100/H100 |
| llama.cpp (4bit) | ~40GB | 中 | 单卡能跑，还要 2-4 张 |
| AirLLM (4bit) | 4GB | 慢 | 没卡硬跑，离线评估 |
| API 调用 | 0 | 极快 | 不差钱 |

**我的观点：** 如果你已经有 GPU（哪怕是 8GB、12GB），用 llama.cpp + GGUF 量化可能是更好的选择，速度比 AirLLM 快几个数量级。AirLLM 真正的受众是那些**连一张过得去的显卡都没有**的人——比如只有一台笔记本或者一张 4GB 入门卡。

但如果你真的要跑 70B+ 模型做正经事，说实话，**充个 API 比 AirLLM 省心一万倍**。Kimi API 或者 DeepSeek API 一次调用几厘钱，不用折腾显卡、不用等两秒一个字。

---

## 真正的坑（你一定会遇到的）

1. **磁盘空间不足**：安装前检查 `df -h`，确保有 200GB+ 空闲
2. **HuggingFace 网关模型**：Llama 等需要 token，别忘了设 `hf_token`
3. **Tokenizer 无 padding token**：代码里设 `padding=False`，否则挂
4. **Windows 兼容性**：官方说支持但实测不够稳定，建议用 WSL2 或 Linux
5. **Mac M 芯片**：需要 MLX 后端，速度比同显存的 NVIDIA 卡还慢
6. **模型拆分时间极长**：第一次加载 70B 模型可能要拆分半小时到一小时

## 总结

| 维度 | 评价 |
|------|------|
| 技术实现 | ⭐⭐⭐⭐⭐ 确实牛逼，一层一层巧妙地绕过了显存瓶颈 |
| 实际可用性 | ⭐⭐ 速度太慢，不适合交互式使用 |
| 性价比 | ⭐⭐⭐ 免费框架，但时间成本很高 |
| 文档 | ⭐⭐⭐⭐ 写得不错，但坑的强调不够 |
| 社区活跃度 | ⭐⭐⭐⭐⭐ 23k stars，issue 回复积极 |

AirLLM 是**一个有趣的技术 demo，但不是一个通用的推理方案**。如果你符合「只有 4GB 卡 + 只做离线评估 + 不赶时间」这三个条件，可以一试。其他情况，要么升级硬件，要么用 API。

**最后一句大实话：** 在 AI 这行，"一张 4GB 显卡跑 70B" 和 "共享单车环游中国" 是一个道理——技术上可行，但正常人不会这么干。
